Manage your AI agent's skills through two types of tools: predefined tools that come built-in with the platform, and custom tools that you can create to extend your agent's functionality.

<Tabs>
  <Tab title="Tools (Predefined)">
    The **Tools (Predefined)** tab allows you to enable or disable built-in capabilities that your AI agent can use during conversations. Each tool represents a predefined function—such as answering FAQs, sending files, or escalating to a human agent—that can be toggled on or off depending on your business needs.  

    When a tool is enabled, additional configuration may be required before the agent can use it successfully. Some tools also depend on the **type of channel** where the end user is interacting (e.g., WhatsApp, Zendesk, Freshchat).  

    ---

    ## How to Enable or Disable a Tool

    1. Navigate to **AI Agent > Design > Skills** in the left menu.  
    2. Under **Tools (Predefined)**, locate the tool you want to manage.  
    3. Use the **toggle switch** to turn the tool on or off.  
    4. If the tool requires configuration, click **Configuration**, complete the required setup, and save your changes.  
    5. Check that the tool is supported by the **channels** your agent is connected to.  

    ---

    ## Available Predefined Tools

    ### **Send Files**  
    - **Description:** Allows the agent to send files directly to the user during a conversation.  
    - **Default:** Always enabled.  
    - **Configuration:** No setup required.  

    ---

    ### **Answer FAQs**  
    - **Description:** Enables the agent to respond to frequently asked questions about your product or service.  
    - **Default:** Always enabled.  
    - **Configuration:** No setup required. You can manage and update FAQs in the **Knowledge** section of the agent.  

    ---

    ### **Collect Information**  
    - **Description:** Enables the agent to collect, verify, and store user information (e.g., name, email, phone number) for lead generation or service requests.  
    - **Toggle:** Off by default.  
    - **Configuration:** Define the fields you want the agent to collect and where the data should be stored.  

    <Note>  
    This tool is **so important** that once enabled, it activates a dedicated **Collect Information tab** in the Skills section. From there, you can create, edit, and manage information collection configurations in detail.  
    </Note>  

    <Accordion title="How to Configure Collect Information">  
    1. Go to **AI Agent > Design > Skills > Collect Information**.  
    2. Click **New Configuration** to start.  
    3. Enter a **Name** for your configuration (e.g., `registro_interesados`).  
    4. Add any **Special Instructions** to guide the agent (e.g., "always verify spelling of the email").  
    5. Define the **fields** you want to collect (name, email, phone, etc.).  
    6. Assign the configuration to one or more **channels** (e.g., WhatsApp, Freshchat).  
    7. Click **Save** to activate the configuration.  
    </Accordion>  

    ---

    ### **Send Notification Email**  
    - **Description:** Sends the conversation details automatically via email after a conversation concludes.  
    - **Toggle:** Off by default.  
    - **Configuration:** Set the recipient email addresses where notifications should be sent.  

    ---

    ### **Human Escalation**  
    - **Description:** Allows the agent to hand off the conversation to a live human agent when it cannot resolve the user's request.  
    - **Toggle:** Off by default.  
    - **Depends on:** A connected integration with **Freshchat** or **Zendesk** is required.  
    - **Configuration:** Set escalation rules and assign the correct integration and group/queue.  

    <Note>  
    This tool only appears when a compatible integration (Freshchat or Zendesk) is connected.  
    </Note>  

  </Tab>

  <Tab title="Custom Tools">

    The **Custom Tools** tab lets you create tools that connect your AI agent to any external API. These tools extend your agent's capabilities beyond the built-in set—you can call CRMs, ERPs, booking systems, proprietary services, or any public API, directly from a conversation.

    ## What are Custom Tools?

    A custom tool is an HTTP request your agent can make during a conversation—based on user input, context, or operator-defined settings. You define:

    - **What** the tool does (name, description)
    - **Where** it calls (URL, HTTP method)
    - **What data** it sends (inputs/parameters)
    - **What comes back** (expected response shape)

    Once saved, the agent decides when to invoke the tool based on the conversation.

    ---

    ## Creating a Custom Tool

    Navigate to **AI Agent > Design > Skills > Custom Tools** and click **Create Tool**. This opens the **Skill Builder**, a step-by-step editor.

    <Tip>
      When you first open the Skill Builder, it loads a **Brevo example** that shows a realistic API integration. You can switch to a different example or start from a blank template using the example selector at the top.
    </Tip>

    ### Step 1 — Name & describe

    Give your tool a clear **Title** and a one-line **Description** explaining when the agent should use it. The description is critical—it's what the AI reads to decide whether to call this tool.

    ### Step 2 — Web link & method

    Enter the **URL** your tool will call. You can include dynamic segments using `{placeholder}` syntax (e.g., `https://api.example.com/users/{user_id}`).

    Select the **HTTP method**: GET, POST, PUT, PATCH, or DELETE.

    ### Step 3 — Inputs

    Inputs are the data your tool needs. Each input has:

    | Setting | Description |
    |---|---|
    | **Name** | The parameter name (used in the request) |
    | **Kind** | Data type: text, integer, decimal, true/false, list, or object |
    | **Source** | Where the value comes from (see below) |
    | **Required** | Whether the agent must always provide this value |
    | **Description** | Helps the agent understand what to fill in |
    | **Allowed values** | Optional comma-separated list to restrict choices |
    | **Location** | Where in the HTTP request the value goes: query string, headers, path, or body |

    #### Input sources

    | Source | Meaning |
    |---|---|
    | **Conversation** | The agent extracts the value from the chat with the user |
    | **Fixed setting** | A constant value the operator sets when configuring the tool (e.g., an API key) |
    | **End user** | Resolved at runtime from the end-user's identity — either their external user ID or a metadata field |
    | **State / Store / Config** | Advanced runtime sources for workflows and internal state |

    #### Object inputs and nested fields

    When you set an input's **Kind** to **object**, a nested field editor appears below it. You can add fields inside the object, each with its own name, kind, description, and options. Objects can be nested multiple levels deep.

    **Per-field source selection (conversation objects)**

    When an object input's source is **Conversation**, each nested field inside it can independently specify where its value comes from. A **"Comes from"** dropdown appears on every nested field with three options:

    | Nested source | Behavior |
    |---|---|
    | **Conversation** | The agent asks the user for this value during the chat. You can set it as required, add a description for the agent, and restrict it to specific allowed values. |
    | **Fixed setting** | The value is a constant set by the operator. A text input appears where you enter the value that will always be sent. |
    | **End user** | The value is resolved at runtime from the end-user's context. You choose between **External user ID** or **A metadata field** (and specify the metadata key). |

    <Tip>
      Per-field source selection is powerful for API calls that need a mix of user-provided data and system-resolved values in the same object. For example, a `contact` object might have the user's `question` filled by the AI (conversation), a `channel` field always set to `"whatsapp"` (fixed), and a `customer_id` pulled from the end-user's external ID (end user).
    </Tip>

    <Note>
      Only fields with their source set to **Conversation** are shown to the AI model. Fixed and end-user fields are injected automatically at runtime—the agent never asks the user for them.
    </Note>

    When the parent object source is **Fixed setting**, all nested fields act as constants (you provide a fixed value for each). When the parent source is **End user**, nested fields are descriptive (schema-only).

    ### Step 4 — Authentication

    If your API requires authentication, add the appropriate headers (e.g., `Authorization: Bearer {token}`) as a fixed-setting input placed in the **headers** location.

    ### Step 5 — What comes back

    Paste an **example JSON response** from your API. The builder uses this to understand the shape of the data that comes back, which helps the agent interpret results for the user.

    Optionally define **output fields** to tell the agent which parts of the response to highlight.

    ---

    ## Saving a Custom Tool

    The **Save** button is enabled once your skill definition is complete:

    - The tool has a **name**
    - The **URL** is a valid web address (starting with `http`)
    - All URL placeholders (e.g., `{user_id}`) have a matching input defined
    - No two inputs share the **same name**
    - A valid **example response** is provided

    If any of these conditions is not met, the Save button is disabled and a message below it tells you exactly what's missing.

    <Note>
      There is no separate test step — the builder validates your configuration inline. Once all fields are complete and valid, you can save immediately.
    </Note>

    ---

    ## Code Preview

    The Skill Builder shows a live **Python code preview** on the right side. This is the actual code that will run when the agent invokes your tool. You can use it to verify the request structure, headers, and parameter placement.

    ---

    ## Editing an existing Custom Tool

    From the **Custom Tools** list, click on any tool to re-open it in the Skill Builder. All fields are pre-populated. Make your changes and click **Save changes**.

    ---

    ## Best practices

    - **Be specific in descriptions** — the agent uses your description to decide when to call the tool. Vague descriptions lead to incorrect invocations.
    - **Use fixed settings for secrets** — never ask the user for API keys. Put them in a fixed-setting input placed in headers.
    - **Test in the Playground** — after saving your tool, go to the Playground and trigger a conversation that should invoke it. Check the Task Log to verify it ran correctly.
    - **Use per-field sources in objects** — when your API expects an object that mixes user input with system values (like user IDs or constant flags), use per-field source selection instead of building separate inputs.
