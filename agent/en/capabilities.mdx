Manage your AI agent's skills through two types of tools: predefined tools that come built-in with the platform, and custom tools that you can create to extend your agent's functionality.

<Tabs>
  <Tab title="Tools (Predefined)">
    The **Tools (Predefined)** tab allows you to enable or disable built-in capabilities that your AI agent can use during conversations. Each tool represents a predefined function—such as answering FAQs, sending files, or escalating to a human agent—that can be toggled on or off depending on your business needs.  

    When a tool is enabled, additional configuration may be required before the agent can use it successfully. Some tools also depend on the **type of channel** where the end user is interacting (e.g., WhatsApp, Zendesk, Freshchat).  

    ---

    ## How to Enable or Disable a Tool

    1. Navigate to **AI Agent > Design > Skills** in the left menu.  
    2. Under **Tools (Predefined)**, locate the tool you want to manage.  
    3. Use the **toggle switch** to turn the tool on or off.  
    4. If the tool requires configuration, click **Configuration**, complete the required setup, and save your changes.  
    5. Check that the tool is supported by the **channels** your agent is connected to.  

    ---

    ## Available Predefined Tools

    ### **Send Files**  
    - **Description:** Allows the agent to send files directly to the user during a conversation.  
    - **Default:** Always enabled.  
    - **Configuration:** No setup required.  

    ---

    ### **Answer FAQs**  
    - **Description:** Enables the agent to respond to frequently asked questions about your product or service.  
    - **Default:** Always enabled.  
    - **Configuration:** No setup required. You can manage and update FAQs in the **Knowledge** section of the agent.  

    ---

    ### **Collect Information**  
    - **Description:** Enables the agent to collect, verify, and store user information (e.g., name, email, phone number) for lead generation or service requests.  
    - **Toggle:** Off by default.  
    - **Configuration:** Define the fields you want the agent to collect and where the data should be stored.  

    <Note>  
    This tool is **so important** that once enabled, it activates a dedicated **Collect Information tab** in the Skills section. From there, you can create, edit, and manage information collection configurations in detail.  
    </Note>  

    <Accordion title="How to Configure Collect Information">  
    1. Go to **AI Agent > Design > Skills > Collect Information**.  
    2. Click **New Configuration** to start.  
    3. Enter a **Name** for your configuration (e.g., `registro_interesados`).  
    4. Add any **Special Instructions** to guide the agent's behavior during data collection.  
    </Accordion>

  </Tab>

  <Tab title="Custom Tools">
    **Custom Tools** let you extend your AI agent by connecting it to external APIs. You define the HTTP request (method, URL, headers, body, etc.) and the agent calls that endpoint when it determines the tool is needed during a conversation.

    ---

    ## Creating a Custom Tool

    1. Go to **AI Agent > Design > Skills** and select the **Custom Tools** tab.
    2. Click **New Tool**.
    3. Fill in the tool details:
       - **Name** — a descriptive label for the tool.
       - **Description** — explain what the tool does so the agent knows when to use it.
       - **HTTP Method** — GET, POST, PUT, PATCH, or DELETE.
       - **URL** — the endpoint the agent will call.
    4. Add **Properties** (parameters) the tool needs.
    5. **Assign channels** where the tool should be available.
    6. Click **Save**.

    ---

    ## Tool Properties

    Each custom tool can have one or more **properties** — the parameters that get sent with the API request. When adding a property, you configure:

    | Setting | Description |
    |---|---|
    | **Key** | The parameter name (e.g., `order_id`, `Authorization`). |
    | **Description** | Explains what the parameter is for. Helps the agent understand when and how to fill it. |
    | **Parameter location** | Where the value is placed in the request — see table below. |
    | **Value type** | The data type: `text`, `number`, `date`, `secret`, or `object`. |
    | **Required** | Whether the property must be present for the request to be sent. |

    ### Parameter location options

    | Location | Description |
    |---|---|
    | **Header** | Sent as an HTTP header. |
    | **Cookie** | Sent as a cookie value. |
    | **Query** | Appended to the URL as a query string parameter. |
    | **Body** | Included in the request body (for POST/PUT/PATCH). |
    | **Path** | Inserted into the URL path (e.g., `/orders/{order_id}`). |
    | **Header Auth Bearer** | Sent as a `Bearer` token in the `Authorization` header. |

    ### Value source

    Each property also requires a **value source** — this tells the platform *where* the value comes from at runtime:

    | Source | Description |
    |---|---|
    | **Configuration** | A fixed value you set when configuring the tool. It stays the same for every request (e.g., an API key or a static parameter). |
    | **Conversation** | The agent extracts the value from the conversation with the user at runtime (e.g., the user provides an order number or email). |
    | **End User** | The value is pulled automatically from the end user's profile data — no need to ask the user or hard-code it. |

    ---

    ### End User value source

    When you select **End User** as the value source, you must specify *which* piece of end-user data to use:

    | End User Source | Description |
    |---|---|
    | **External User ID** | Uses the end user's unique external identifier (the ID passed by the channel or set via the API/Playground). |
    | **Metadata** | Uses a specific metadata key from the end user's profile. When you select this option, an additional **Metadata Key** field appears where you enter the exact key name (e.g., `plan`, `company_id`, `language`). |

    <Tip>
    The **End User** value source is ideal for personalizing API calls without asking the user for information they've already provided — for example, passing their account ID or subscription plan to an external system automatically.
    </Tip>

    <Note>
    End-user data (external user ID and metadata) comes from how the user was identified on the channel. You can test this in the **Playground** using the **External User** configuration. See the [Playground docs](/agent/en/playground) for details.
    </Note>

    ---

    ### Assigning channels and per-channel end-user mapping

    When you assign a custom tool to channels, you can configure how **End User** properties are resolved on each channel. Because different channels may store user information differently, the platform lets you map each end-user property to the correct source on a per-channel basis.

    ---

    ## Key notes

    - Custom tools are executed **server-side** — the agent decides when to call them based on the conversation context and the tool's description.
    - Properties marked as **secret** are encrypted and never exposed in logs or conversation history.
    - You can **edit or delete** custom tools at any time. Deleting a tool removes it from all assigned channels.
